# Command Palette: prefix-routed Commands from Providers in a lifetime-scoped Registry

The Command Palette (CONTEXT.md) treats Entity/World search results and static actions uniformly as **Commands**, routed by a typed prefix the way VSCode's Quick Open (no prefix) and Show Commands (`>`) are — rather than rendering search results and actions as two different UI sections. Each **Command Provider** declares the prefix(es) it responds to and owns its own matching (a server-side Entity search, a client-side filter over the already-loaded World list, a static list — no shared fuzzy-match engine). Prefixes are many-to-one: several Providers can answer the same prefix, and the Palette merges their results into stable, provider-ordered sections so a slower server-backed Provider filling in late never reorders one that resolved instantly.

Providers register into one root-level **Command Registry** (`register(provider): () => void`). Built-in Providers (Entity quick-open, World quick-open, Create Note, Create Map) register once at app bootstrap and live for the app's lifetime; a future contextual Provider (e.g. a map tool-switcher) registers from its owning component's constructor and unregisters on destroy — the same `onCleanup`/`takeUntilDestroyed` idiom already used for scoped effects elsewhere (e.g. the Inspector's Entity Link). The Registry does not distinguish "built-in" from "contextual" — both are just Providers with different registration lifetimes — so contextual Commands (map-only tool shortcuts, the concrete driver for this decision) slot in later without reshaping the Palette itself.

The Palette overlay itself reuses the existing `Dialog` primitive (native `<dialog>`, ADR-0007) for the modal shell; the keyboard-navigable, multi-section result list is a new component rather than an extension of the existing `SuggestionMenu` (which is coupled to the `@tiptap/suggestion` plugin's single-list lifecycle and doesn't fit multi-Provider, prefix-switching results). The Entity quick-open Provider needs no new server endpoint — it searches globally (not scoped to the active World) via the existing paginated `q`/`ids` read surface (ADR-0025).

## Considered Options

- **Static command list + a context predicate** (e.g. `context: 'map'`) instead of component-registered Providers. Rejected: requires a context enum kept in sync by hand across every feature, and doesn't self-cleanup on navigation the way a component's own lifecycle does.
- **One Provider per prefix (prefix as a unique key).** Rejected in favor of many-to-one: it would force Entity quick-open and World quick-open into a single Provider that knows about both domains, instead of two small, independent ones that both happen to answer the empty prefix.
- **A shared client-side fuzzy-match utility all Providers must use**, for consistent ranking. Rejected: Providers legitimately need different matching strategies (server search vs. client filter vs. a static list), and ADR-0025 already deferred building a shared matching/ranking engine until a real need forces it.
- **Model search results as a distinct, non-Command type**, rendered in a separate UI section from actions. Rejected in favor of one uniform Command model routed by prefix — the explicit goal was VSCode-like behavior, where Quick Open and Show Commands are the same list mechanism under a different prefix, not two different widgets.

## Consequences

- Any future contextual Command (map tool-switching being the concrete example) is just a new Provider registered from that feature's own component — no change to the Palette, the Registry, or existing Providers.
- "Create Note" / "Create Map" are global built-in Commands (always listed under `>`, not gated on an active World), opening a dedicated create dialog with name and World (defaulting to `activeWorld() ?? worlds()[0]`) — a separate, more explicit flow from the Inspector's existing inline create-and-link-from-typed-query, not a replacement for it.
- Because prefixes are many-to-one and each Provider owns its own matching, adding a new no-prefix or `>` source later (e.g. recent Entities, or a future symbol-style prefix) never requires touching another Provider.
