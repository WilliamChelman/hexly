# Entities autosave: a debounced, single-flight save driven by reference-equality dirty tracking

The Save button is gone from every Entity surface (`note` and `hexmap`). Edits to the savable payload — grid, Content, Tags — now persist automatically through `EntitySession`: an **800ms trailing debounce** arms a save whenever the Entity is dirty. **Dirty is derived, not signalled** — a `computed` that compares the three live references (`HexMapStore.document()`, `content()`, `tags()`) against a baseline captured on load and reset after each clean save. Reference equality is sound *because* of existing discipline (ADR-0005): immer's `commit`/`undo`/`redo` yield a new doc reference only on a real change, `setContent` is minted fresh from TipTap's `update` event (real edits only), and `setTags` sets a fresh array. The point of deriving rather than firing explicit edit events: a future editing widget **cannot forget to mark dirty** — if it mutates any savable signal, dirty already sees it. The only contract is that a genuinely new savable field must join the baseline tuple and the save payload — enforced by the shape of `save()`, not by remembering to wire a notifier.

The scheduler is **single-flight with coalescing**: never two saves in flight; if edits land during an in-flight save, exactly one follow-up fires afterward, re-read off `dirty()` so it sends the newest snapshot (no edit is lost mid-save). It is **gated on `!conflict()`** — a conflict doesn't advance the version, so an ungated scheduler would hammer the server every 800ms with the same stale base version; instead autosave pauses, edits accumulate locally, and the existing conflict chip + manual Reload resolves it (ADR-0019, issue #6). The Save button's two old jobs — feedback and error/retry — move to a single `aria-live="polite"` **status chip** with five states (`Edited elsewhere` → `Save failed` (Retry) → `Saving…` → `Unsaved…` → `Saved`), shared by both surfaces. **Cmd/Ctrl+S is retained** as a manual flush (and to suppress the browser's "save page" dialog).

Persistence on the way out is **best-effort, layered**: the route-scoped session flushes a pending save on `destroyRef.onDestroy` (the `HttpClient` request outlives the torn-down component), and a `beforeunload` prompt guarded on `dirty()` covers tab-close / refresh / external navigation. Neither path can surface a conflict or error — the UI is already gone — and that's accepted: last-write-wins means a stale exit-save lost anyway, and the in-app chip covers the case where the user stays. **Rename is untouched**: the server already isolates it (PATCH writes only `name`, never bumps `version`; PUT writes `document/version/tags`, never `name`), so it neither races the body save nor needs to join the channel.

## Considered Options

- **Explicit `markEdited()` notifier called at each edit chokepoint.** Rejected: it's a wiring contract, and the stated risk is exactly that a future widget mutates state without calling it — silent unsaved changes. Deriving dirty from the payload removes the contract.
- **Deep-equality dirty tracking.** Rejected as unnecessary given ADR-0005's immer discipline and TipTap's mint-on-update: reference equality is correct here and avoids per-keystroke deep compares of large grid/Content documents. The linchpin guarding this is the test that an edit during an in-flight save survives (baseline resets to the *sent* snapshot, not the live one).
- **`navigator.sendBeacon` on `pagehide` for unload persistence.** Rejected for now: the save is a version-checked PUT returning JSON for conflict handling; a beacon is response-less and would need a new endpoint shape. Over-engineering at ~5 users (ADR-0019's own framing) until real loss is reported — the `beforeunload` prompt is the cheap standard guard.
- **Auto-merge or auto-reload on conflict.** Rejected: last-write-wins + manual Reload is the established contract (ADR-0019); conflicts are rare in the single-user norm and silent resolution risks discarding work without consent.

## Consequences

- e2e/specs keyed on `data-testid="save"` adapt to assert on the status chip instead; there is no Save button on any surface.
- A network save failure pauses the scheduler (dirty is unchanged, so it can't self-retry into a loop); recovery is the chip's Retry link, the user's next edit re-arming the scheduler, or the destroy/`beforeunload` guards. No automatic backoff — add it only if flaky-network loss becomes a real complaint.
- The Map↔Note view toggle needs no flush: the session survives the toggle and Content/grid live in session/store signals, so the recreated editor reseeds from live state without touching dirty.
- Turning on real-time collaboration later (ADR-0019, deferred) would replace this scheduler wholesale with CRDT sync; the reference-equality dirty model is a single-writer construct and is not a step toward it — but it's contained in `EntitySession` and cheap to discard.
