# Entities autosave: a debounced, single-flight save driven by reference-equality dirty tracking

The Save button is gone from every Entity surface (`note` and `hexmap`). Edits to the savable payload — grid, Content, Tags — now persist automatically through `EntitySession`: an **800ms trailing debounce** arms a save whenever the Entity is dirty. **Dirty is derived, not signalled** — a `computed` that compares the three live references (`HexMapStore.document()`, `content()`, `tags()`) against a baseline captured on load and reset after each clean save. Reference equality is sound *because* of existing discipline (ADR-0005): immer's `commit`/`undo`/`redo` yield a new doc reference only on a real change, `setContent` is minted fresh from TipTap's `update` event (real edits only), and `setTags` sets a fresh array. The point of deriving rather than firing explicit edit events: a future editing widget **cannot forget to mark dirty** — if it mutates any savable signal, dirty already sees it. The only contract is that a genuinely new savable field must join the baseline tuple and the save payload — enforced by the shape of `save()`, not by remembering to wire a notifier.

The scheduler is **single-flight with coalescing**: never two saves in flight; if edits land during an in-flight save, exactly one follow-up fires afterward, re-read off `dirty()` so it sends the newest snapshot (no edit is lost mid-save). It is **gated on `!conflict()`** — a conflict doesn't advance the version, so an ungated scheduler would hammer the server every 800ms with the same stale base version; instead autosave pauses, edits accumulate locally, and the existing conflict chip + manual Reload resolves it (the optimistic-`version`/409 + re-pull contract of ADR-0002/0005, issue #6). The Save button's two old jobs — feedback and error/retry — move to a single `aria-live="polite"` **status chip** with five states (`Edited elsewhere` → `Save failed` (Retry) → `Saving…` → `Unsaved…` → `Saved`), shared by both surfaces. **Cmd/Ctrl+S is retained** as a manual flush (and to suppress the browser's "save page" dialog).

Persistence on the way out is **awaited for in-app navigation, best-effort for tab teardown**. `EntitySession.flush()` returns an Observable that drains any in-flight autosave, sends the latest pending snapshot, and completes when it lands; it is **awaited** by `openRoute` (an Entity→Entity swap on the same route — a `:id` param change, not a deactivation) and by a **`CanDeactivate` guard** (leaving `/entities/:id` entirely). So an in-app leave can't drop a debounced edit, and a mid-save edit rides a coalesced follow-up under the advanced version. The cost is a save round-trip of navigation latency when leaving dirty; flush is bounded by `FLUSH_TIMEOUT_MS` so a hung network can't trap the user. **Tab-close / refresh / external navigation** stays best-effort: a `beforeunload` prompt guarded on `dirty()` lets the user stay and let autosave finish (the page is gone before async work can land, and a beacon was rejected below). The awaited paths still can't usefully surface a conflict/error — the user is leaving — and that's accepted under last-write-wins. **Rename is untouched**: the server already isolates it (PATCH writes only `name`, never bumps `version`; PUT writes `document/version/tags`, never `name`), so it neither races the body save nor needs to join the channel.

## Considered Options

- **Explicit `markEdited()` notifier called at each edit chokepoint.** Rejected: it's a wiring contract, and the stated risk is exactly that a future widget mutates state without calling it — silent unsaved changes. Deriving dirty from the payload removes the contract.
- **Deep-equality dirty tracking.** Rejected as unnecessary given ADR-0005's immer discipline and TipTap's mint-on-update: reference equality is correct here and avoids per-keystroke deep compares of large grid/Content documents. The linchpin guarding this is the test that an edit during an in-flight save survives (baseline resets to the *sent* snapshot, not the live one).
- **`navigator.sendBeacon` on `pagehide` for unload persistence.** Rejected for now: the save is a version-checked PUT returning JSON for conflict handling; a beacon is response-less and would need a new endpoint shape. Over-engineering at ~5 users (ADR-0019's own framing) until real loss is reported — the `beforeunload` prompt is the cheap standard guard.
- **Auto-merge or auto-reload on conflict.** Rejected: last-write-wins + manual Reload is the established contract (optimistic `version`/409 from ADR-0002, last-write-wins from ADR-0004, client re-pull from ADR-0005, generalized per-Entity by ADR-0018; ADR-0019 only defers real-time collaboration on top of it); conflicts are rare in the single-user norm and silent resolution risks discarding work without consent.

## Consequences

- e2e/specs keyed on `data-testid="save"` adapt to assert on the status chip instead; there is no Save button on any surface.
- **Revises ADR-0022's header composition.** ADR-0022 describes the editor header as projecting an "Editing/Conflict chip into `title` and Save/Share into `actions`". The Save button is gone and the Editing/Conflict chips are replaced by the single {@link SaveStatus} chip; Share stays. ADR-0022's slotted-header decision is otherwise intact.
- A network save failure pauses the scheduler (an `_error`-gated arm condition; it stays paused until the payload changes or a manual Retry, so it can't self-retry into a loop); recovery is the chip's Retry link, the user's next edit re-arming the scheduler, or the leave-flush. No automatic backoff — add it only if flaky-network loss becomes a real complaint.
- **In-app navigation away from a dirty Entity blocks on a save round-trip** (the awaited flush), bounded by `FLUSH_TIMEOUT_MS`. Accepted as the price of never silently dropping an edit; if the latency is ever felt, the fallback is to drop back to fire-and-forget for the swap and keep the await only on full route-leave.
- The Map↔Note view toggle needs no flush: the session survives the toggle and Content/grid live in session/store signals, so the recreated editor reseeds from live state without touching dirty.
- Turning on real-time collaboration later (ADR-0019, deferred) would replace this scheduler wholesale with CRDT sync; the reference-equality dirty model is a single-writer construct and is not a step toward it — but it's contained in `EntitySession` and cheap to discard.
