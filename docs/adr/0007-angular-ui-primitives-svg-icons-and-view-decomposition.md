# UI primitives as Angular components, SVG icons as components, and decomposed view shells

Reusable UI is expressed as **Angular primitives that own their styles** — not as global CSS component classes applied by hand. The global component-class layer (`apps/web/src/styles/components.css`) is **eliminated**: every class in it (`.btn`, `.panel`, `.tool`, `.field`, `.input`, `.chip`, `.coord`, `.eyebrow`, `.kbd`, `.swatch`, `.dot`, …) becomes a primitive. SVG glyphs live in **their own components**, never inlined into feature templates. Large view shells are **decomposed** into one component per region rather than one file rendering the whole screen.

This **amends how ADR-0006 is applied**, not its core. ADR-0006's token layer stands unchanged: colours, type, spacing, radii, and terrain fills remain CSS custom properties on `:root`/`[data-theme]`, because the Canvas renderer (ADR-0003) paints with `var(--terrain-forest)` and a component cannot reach that surface. What changes is the *second half* of ADR-0006 — the shared **global component classes**. Those manufactured a flat global namespace and were applied as string soup (`class="btn btn--primary btn--sm"`); in a component framework that work belongs to typed primitives, and the namespace they existed to provide is made redundant by view-encapsulation.

> Supersedes this ADR's own first iteration, which kept native-element widgets as **directives mapping onto the global classes** (on the reasoning that a directive can't own styles, so the classes had to stay). That compromise is dropped: see the Decision below.

## Considered Options

**Keep global component classes + BEM in component styles.** The status quo from ADR-0006. Rejected for three reasons that surfaced in review:

1. **The classes want to be primitives.** `.btn`/`.chip`/`.coord` are reusable UI, and the framework's unit of reuse is a directive or component, not a memorised class string. A typed `appButton` with `variant`/`size`/`danger` inputs is discoverable, refactor-safe, and can't be mis-spelled.
2. **BEM in `styles` is redundant.** Angular view-encapsulation already scopes a component's CSS (`[_ngcontent-…]`). The `block__element--modifier` ceremony only earns its keep in a *global* sheet; inside an encapsulated component `.shell__header` buys nothing over `.header`. Where BEM-density felt necessary to navigate a template, that was a **size smell**, not a need for namespacing.
3. **Inlined SVG sprites bloat templates.** A 180-line `<defs>` sprite and `<use href="#g-…">` references made `editor-shell` hard to read and impossible to reuse a glyph elsewhere without copying the sprite.

**Wrapper components for everything (`<app-button>`).** Rejected for native-element widgets. Wrapping a `<button>`/`<input>` in a host *element* (`<app-button><button>…`) loses or duplicates the real element's type, form participation, focus, and a11y. The attribute-selector component below avoids this — the host *is* the native element.

**Attribute directives mapping onto global classes.** This ADR's first cut. A directive keeps the native element but **cannot own styles**, so the `.btn*`/`.panel*` classes had to remain global, and `components.css` lived on. Rejected on reflection: it leaves the exact global-class layer this ADR set out to remove, just with a typed front door.

## Decision

- **Every widget → a component with an attribute selector on its native element.** `@Component({ selector: 'button[appButton], a[appButton]', template: '<ng-content/>', styles: … })`. The host stays the real `<button>`/`<a>`/`<input>`/`<section>` (full type, focus, form, a11y, semantics); `<ng-content>` projects its children; and **the component owns its scoped styles** — so the global class is deleted, not merely fronted. Typed `input()`s drive variants, mapped to `.is-*` host classes (`variant="primary"` → `:host(.is-primary)`). Widgets whose internal parts need styling (e.g. `appTool`'s swatch/label/hint) render that structure from inputs so encapsulation reaches it. `components.css` is removed entirely.
- This holds even for atoms (`appEyebrow`, `appCartouche`, `appSwatch`, `appDot`, `appKbd`, `appRule`, `appGlyphBox`): the goal is that no global component-class layer survives. The token custom properties (ADR-0006) are the only shared CSS that remains, because the Canvas renderer needs them. An atom shared by two regions (e.g. `appGlyphBox`, the leading icon tile used by both `appTool` and the inspector header) becomes a primitive rather than a copied local class; a parent may still restyle its host from the outside (the tool's active state recolours `[appGlyphBox]`).
- **SVG glyphs → one component each** (`app-icon-*`), one per file under `ui/icon/glyphs/`. A consumer **imports the glyph components it needs** and references them by selector (`<app-icon-share [size]="16" />`); where the glyph is chosen at runtime (e.g. the theme toggle's sun/moon, a tool's content glyph) the consumer selects locally with `@if`/`@switch` over the small set it can show. There is **no `app-icon` dispatcher** — a single name-keyed component would have to import every glyph, pulling the whole set into every consumer; importing directly keeps each consumer's glyph dependency explicit and tree-shakeable. Nothing inlines an icon's markup into a feature template. **Exception:** glyphs drawn *inside* an SVG illustration (the map's feature markers) can't be HTML components — SVG-in-SVG crosses the namespace — so they live as a `<defs>` local to that illustration's own component (`MapCanvas`). The illustration is itself "the SVG in its own component."
- **View shells → one component per region.** `editor-shell` becomes a layout orchestrator over `app-editor-header`, `app-tool-palette`, `app-map-canvas`, `app-inspector`, `app-status-bar`. Each owns its own markup, data, and short structural class names — so the `shell__`/`tools__`/`inspector__` BEM prefixes disappear by construction.
- **Selector prefix is `app`** (per the workspace eslint `@angular-eslint/*-selector` rules), hence `appButton` and `app-icon-*` rather than a `hexly`-prefixed name. Renaming the workspace prefix is a separate decision.

Primitives currently live under `apps/web/src/app/ui/`. They may graduate to a `libs/ui` Nx library once a second app consumes them; that move is deferred until there's a real second consumer.

## Consequences

- There is **no global component-class layer** — `components.css` is gone. A new widget is a new primitive; there is no "just add a class" path, which is the point. The only shared CSS is the token layer.
- A primitive that styles internal parts must **render them from inputs** rather than projecting them, because emulated encapsulation won't style projected content. `appTool` therefore takes `label`/`hint`/`swatch`/`glyph` inputs instead of accepting arbitrary children — a deliberate constraint, not an oversight.
- Reviewers should read a region's component, not a 700-line shell. Armed-tool state is owned by `ToolPalette` for now (the canvas is a frozen placeholder per ADR-0003); when painting is wired (ADR-0005) it graduates to an editor-state service shared by palette and canvas.
- A glyph used both as an HTML icon and as an in-map marker (e.g. *settlement*) is defined twice — once as an `app-icon-*` component, once in the `MapCanvas` `<defs>`. Accepted: the map SVG is a placeholder, and the duplication is local and small.
