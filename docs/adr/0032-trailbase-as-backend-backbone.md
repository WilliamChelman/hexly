# TrailBase as the backend backbone

Hexly's server is a hand-written NestJS app over SQLite/Drizzle: custom argon2 + opaque-session auth (ADR-0004), an owner-scoped `EntitiesService` (the World-cascade and entity-grant sharing in `CONTEXT.md` is in fact **not yet enforced** in code), drizzle-kit migrations at boot (ADR-0027), and the SPA served from the same origin (ADR-0008). We want, over time: enforced ACL, an admin role, file handling, and **opt-in realtime sync** — and realtime is the real driver, because it dictates the client's contact layer ("hot entity" vs fire-and-forget HTTP) and so must be decided before more client is built. We will **replace the backbone with [TrailBase](https://trailbase.io)** (single Rust binary: SQLite + extensions, Record APIs with ACL/SQL access-rules, built-in auth + admin UI, SSE realtime subscriptions, file storage, a WASM custom-endpoint runtime). The migration happens **pre-v1, before any real user data exists** — a backbone swap is dramatically cheaper now than after non-technical friends are relying on the app, and post-v1 it would demand a smooth automated data migration we'd rather never write.

## Decisions

- **Entities are a TrailBase Record API**, not a hand-written service. This is forced by realtime: TrailBase's SSE change-feed is bound to the Record API, so to get a live `entities` you must expose `entities` through it. Consequences:
  - **Validation moves off the write path.** TrailBase has **no middleware** — custom endpoints are standalone WASM routes that cannot intercept Record API writes. So `@hexly/domain` zod becomes the **client-side** validator and the source for a server-side `CHECK(jsonschema(...))` backstop. zod no longer runs *in* the write path (despite the earlier assumption that a "TS middleware" existed — it does not).
  - **Optimistic concurrency** (`version`, today a 409-with-current-state) becomes an **UPDATE access-rule** `_REQ_.version = _ROW_.version`, which rejects a stale write. The 409 *body* stops mattering because a subscribed client already holds current state via SSE.
  - **Invariants go declarative:** "Home can't be deleted" → DELETE rule `_ROW_.is_home = FALSE`; "at most one Home per World" → the existing partial unique index; descriptor harvest (ADR-0023) and "creating a World also creates its Home Entity" (ADR-0024) → SQLite `AFTER INSERT` triggers. **WASM is held in reserve for *writes*** — TrailBase has no write middleware and the jsonschema `CHECK` + version rule + triggers cover the write path, so we add no custom write code for v1. (For *reads* the declarative layer genuinely can't express — a user-context-gated aggregate like a visibility-gated `DISTINCT` — WASM is a first-class tool, not a near-never last resort: **ADR-0034 relaxes this stance**, and the descriptor-vocabulary read is the first such endpoint.)
- **Auth is TrailBase's** (short-lived JWT + opaque refresh token), because Record API access-rules key off `_USER_.id` from TrailBase's auth context — keeping our own auth would blind the ACL. The **closed user set** of ADR-0004 is preserved by **disabling public registration** and provisioning users as an admin; this also gives the wanted **admin role + admin UI for free**.
- **`apps/api` (NestJS) is deleted.** TrailBase is the sole server for v1 and serves the built Angular SPA as static assets, so the single-origin of ADR-0008 holds.
- **Heavy/stateful future layers are additive sidecars, never a backbone swap.** Markdown import/export and map screenshots are doable in pure TrailBase (client-side, or a WASM route). True **Yjs co-editing (Hocuspocus) cannot** run in the WASM sandbox and is a separate Node service when it lands — but adding it touches no TrailBase data or auth, so it doesn't reintroduce the post-v1 migration risk we're avoiding.
- **The "hot entity" client seam abstracts over realtime providers.** There are two realtimes: TrailBase SSE (last-write-wins, whole-document broadcast — available v1, no extra service) and Yjs CRDT (merge, sidecar, later). The client's hot-entity contact layer is designed so a Yjs provider can slot in behind the same interface without a third client rewrite. (Reinforces ADR-0019's "co-editing deferred, not precluded.")

## Considered Options

- **Keep NestJS, build ACL/admin/files/realtime by hand.** Rejected: we're already on SQLite, so TrailBase's only real cost vs. this is the migration itself — and building all four batteries by hand is strictly more code for a passion project whose point includes *learning TrailBase*.
- **TrailBase behind a thin NestJS gateway** (client → Nest → TrailBase). Rejected: once auth and entities live in TrailBase, the gateway has almost nothing to do, and it would re-add the very layer we're deleting.
- **Entities behind custom WASM routes** (keep zod + 409 in the write path). Rejected: forfeits the realtime subscription on entities, which is the whole reason for the move, and forces hand-rolled access checks.
- **Migrate after v1.** Rejected: real data from non-technical users turns a reseed into a risky automated migration.

## Consequences

- **The server stops being a domain-shaped API and becomes "a database with rules + (eventually) a few WASM routes."** The client talks to generic Record API endpoints. This is a real identity change for the codebase.
- **Supersedes / amends prior ADRs.** ADR-0027 (drizzle-kit migrations at boot) is **superseded** — TrailBase owns migrations: timestamped SQL files in `traildepot/migrations/` (scaffolded by `trail migration`), applied at startup with an apply-once ledger, append-only/forward-only. This meets every ADR-0027 requirement (ordered, versioned, no manual step for non-technical self-hosters), so `drizzle-kit`, `drizzle-orm`, and `schema.ts` are dropped — fewer moving parts, not more. TS-logic backfills (ADR-0027's deferred concern) become a one-off Record-API script or WASM routine, added when first needed. Amended by reference here, not edited in place: ADR-0004 (auth → TrailBase JWT/refresh + TTL-bounded revocation; sharing → declarative access-rules), ADR-0008 (TrailBase serves the SPA), ADR-0002 (entities are now a Record API table, still one JSON document per row), ADR-0024 (the World/visibility cascade is now access-rule SQL), ADR-0019 (co-editing = Yjs sidecar). Amended by ADR-0034 (WASM custom endpoints own reads the Record-API layer can't express — user-context-gated aggregates; the descriptor vocabulary is the first, relaxing the "WASM in reserve" tone above).
- **Revocation weakens** from instant (delete the session row) to TTL-bounded (revoke the refresh token; the live JWT survives until it expires). Accepted.
- **Session storage moves from an HttpOnly cookie to `localStorage`** (the web client holds TrailBase's JWT + refresh token). This is a real XSS-exposure downgrade from ADR-0004's HttpOnly cookie — script on the origin can read the tokens. It is **forced by keeping Hexly's own login screen** (ADR-0022): TrailBase's JSON login is token-only (`response_type` ∈ {`token`, `code`}), and its HttpOnly-cookie session is reachable *only* via the hosted auth UI redirect flow, which would replace our login page. Accepted, with the mitigation being a strict **Content-Security-Policy** (the actual defence against token theft) plus the short auth-token TTL; revisit if we ever adopt the hosted auth UI. `sessionStorage` (narrower window, but no cross-restart persistence) was considered and rejected to preserve the prior "stay signed in" UX.
- **Open items for follow-up grilling**, not decided here: the hot-entity client contact-layer design (when a subscription opens, autosave × SSE-broadcast interaction, the exact provider interface Yjs slots into); and file-column specifics. Disabling public registration is **confirmed supported**. The pre-v1 data port is a trivial reseed.

## Spike validation (#127)

A throwaway TrailBase v0.29.0 spike validated the three risky assumptions over the real HTTP Record API. **All confirmed:**

- **`version` UPDATE access-rule** (`_REQ_.version = _ROW_.version`) rejects a stale save (HTTP 403) and admits a current one. The stored counter is advanced by an `AFTER UPDATE` trigger (recursion-safe via `WHEN NEW.version = OLD.version`), not by the rule itself — the client sends the base version it last read.
- **`AFTER INSERT ON worlds` trigger** mints exactly one Home Entity (`is_home = 1`, title = World name); it composes with the `entities` jsonschema `CHECK`. The `idx_world_home` partial unique index and the `is_home = 0` DELETE rule are both in force.
- **`CHECK(jsonschema(z.toJSONSchema(entityBodySchema)))` is sufficient — WASM is NOT required (this open assumption is resolved).** Zod 4 emits a `oneOf` of the `note`/`hexmap` arms (`const` discriminants, `additionalProperties:false`, `enum`, `pattern`); TrailBase's SQLite `jsonschema_matches` enforces it and rejects malformed bodies (bad discriminant, missing `content`, bad enum, extra property) at the DB layer (HTTP 400). Both Zod's default `draft 2020-12` output and `{target:'draft-7'}` are accepted.

Implementation notes: prefer registering the domain schema **by name** in `config.textproto` (`jsonschema('name', col)`) over embedding it inline; Record-API PKs must be UUID (`CHECK(is_uuid_v7(id))`) or INTEGER; composite PKs are unsupported (use a surrogate id + `UNIQUE`); `autofill_missing_user_id_columns` fills `owner_id` from the caller. The World-cascade share read-rule was also validated end-to-end with a second user. Full findings on #126.
