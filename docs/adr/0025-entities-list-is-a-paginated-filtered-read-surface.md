# The entities list is a paginated, filtered, enveloped read surface; the client query/cache layer is deferred

`GET /api/entities` stops returning a bare, unbounded `EntitySummary[]` and becomes a filtered, cursor-paginated read surface returning an envelope `{ items: EntitySummary[], nextCursor: string | null }`, with optional query params `ids`, `q`, `type`, `cursor`, `limit`. Summaries only (no `document` body), as today. The driver is three things on the horizon — Obsidian vault import (1000+ entities), faceted/fulltext search, and visibility of not-owned entities — none of which need building now, but all of which break if consumers keep assuming "fetch the whole list." **The contract is the guardrail**: the envelope makes a full-list fetch a deliberate, visible act (loop the cursor) rather than the default, so future consumers don't bake the assumption in. The matching engine (real search, faceting, cross-owner queries, the import) is deferred — only the shape changes now.

Paging is **mandatory**, not lazy-default: the response is always an envelope, even page one. The cursor is **opaque** to clients (they only pass `nextCursor` back); the server encodes it as a trivial offset today and can swap to a keyset/ranking cursor when search lands **without touching any consumer**. No `total` and no jump-to-page — load-more / infinite scroll is the honest UX for an entity list and is what the picker wants too. `ids` is just another filter on this one endpoint (not a new route): the inspector's Entity Link picker passes its one linked id to resolve a name without pulling the target's whole document via `load(id)`, and ADR-0023's future inline-link resolver passes many ids per note through the same path. `load(id)` (`GET /entities/:id` → `EntityDetail`) stays what it is — opening an Entity to edit its document.

## Considered Options

- **Keep the bare unbounded array, fix only the picker's DOM render.** The observed symptom was render cost, not data. Rejected: it leaves the contract assumption (full-list fetch) in place for every future consumer — the import and search work would then have to unwind it. Fixing the shape once is the smaller total change.
- **Envelope with a large lazy-default page (no behaviour change for today's consumers).** Rejected in favour of mandatory paging: the entity-browser is a list/search page that wants real pagination anyway, and the picker's needs (server-side text search to pick, `ids`-resolve to display) are straightforward — so there's no consumer the lazy default actually spares.
- **`offset`/`limit` + `total`, numbered pages.** Cheap over a name-sorted SQLite table, but `total` and stable offsets get expensive and semantically fuzzy over ranked, faceted fulltext results — exposing them now would force the search feature to fake them or break consumers. The opaque cursor avoids committing the contract to offset semantics.
- **Adopt `@tanstack/angular-query-experimental` now.** Rejected: it's pre-stable (still `-experimental` as of 5.101.1, ships breaking changes in minor/patch releases), and we'd be betting a core data path on it.
- **Build an in-house Angular-signals layer over the stable `@tanstack/query-core`.** Investigated. `query-core` is stable and zero-dep, but the signals glue we'd own (lazy observer, options-via-effect, subscribe-outside-zone, teardown, SSR) is ~150–300 LOC that is a strict *subset* of the experimental adapter TanStack already maintains — i.e. reinventing it minus the maintenance. Only one of three consumers (ADR-0023's unbuilt inline resolver) actually needs a cache. Rejected as premature: Angular 21's native `httpResource()` + `linkedSignal` covers today's two consumers with no dependency and no glue.

## Consequences

- The two current consumers adapt to the envelope: the **entity-browser** drives the cursor list (load-more) and reloads after create/rename/delete; the **Entity Link picker** uses `q` for server-side search when picking and `ids` to resolve the linked name for display. Both via native `httpResource`/signals on the existing `EntitiesClient` — no new data layer.
- **No client query/cache library now.** The trigger that flips this: ADR-0023's inline id→name resolver is actually being built **and** real duplicate-resolve / cross-consumer invalidation pain is observed. Even then the first move is a ~20-line `Map<id, EntitySummary>` batch cache; only if that's insufficient do we adopt the official `@tanstack/angular-query-experimental` — not a hand-vendored core layer.
- The opaque cursor is the load-bearing forward-compat choice: search/faceting/cross-owner visibility change the cursor's *internal* encoding and the query behind it, not the wire contract.
- This does not build search, faceting, cross-owner queries, or the importer. It only ensures those land against a contract that already bounds and filters, instead of one that promises the whole list.
